# The diagnostics standard — rules, then an audit

Status: **ACCEPTED 2026-07-16** — all four open questions signed off by the
user: (1) `Error` gains the secondary note; (2) parser errors stay generated
until H6, targeted labels only; (3) warnings are in scope, same ledger;
(4) the ledger is a tracked proposal file. The audit proceeds in batches. The v0.8.0 arc, as set by
the user: first define the rules for **where** a diagnostic anchors and
**what** it says; then test every diagnostic against them and fix the ones
that fail. Survey: **181 `diagnostics.push` sites** (155 analyzer, 22 macro
engine, 2 async inference, 2 platform coloring) plus the generated
lexer/parser errors (their own class, §4); **105 `assert_fails_spanning`
pins** already stand.

Prior art this codifies: E7's first pass ("the narrowest expression that
identifies the problem; call-argument mismatches are the model"), E8's
boundary sketch, and this week's diagnostics (the B13 origin-naming, the
`.map_err`/`.ok_or` steers, the condition-check wording).

## 1. Anchoring rules — WHERE

- **A1 — Narrowest identifying span.** The span covers the smallest
  expression that identifies the problem, not the enclosing statement:
  the mismatched *argument*, not the call; the offending *field value*,
  not the initializer; the failing *path segment*, not the whole import.
- **A2 — User code only.** A diagnostic caused by user code never anchors
  in `std` or generated source. It walks back to the nearest user-written
  anchor — the annotated item, the call, the macro attribute that produced
  the code — and leads with that; the internal frame may follow as a
  secondary note (§3's mechanism), never as the primary span.
- **A3 — Order-independent.** The same program produces the same anchor
  regardless of declaration order, import order, or fixpoint scheduling.
- **A4 — The anchor catalog.** The specific rule per class, so an audit has
  a checklist: argument mismatch → that argument; struct-field mismatch →
  the field's value; match-leg type conflict → the leg's body; name/path
  failure → the failing segment; operator mismatch → the operator site,
  naming both operand types; pattern arity → the pattern; condition → the
  condition expression; annotation conflict → the value (the annotation is
  the contract, the value broke it).

## 2. Content rules — WHAT

- **B1 — User vocabulary only.** Rendered types and names as the user
  writes them. Internal terms — slot, binder, constraint, entity, world,
  monomorphization — never appear.
- **B2 — Expected and got, both rendered.** The house form stands:
  "Expected X, but got Y instead." A diagnostic that names only one side
  fails the rule.
- **B3 — Name the invisible decision.** When inference chose something the
  user never wrote, a conflict must say where the choice came from — the
  model is B13's "inferred from the closure's FIRST call". If the compiler
  decided, the compiler says so.
- **B4 — Steer when the fix is unambiguous.** One concrete action, code-
  shaped when short: `.map_err(…)` / `.ok_or(err)` / "annotate the binding
  (e.g. `: Map<str, i32>`)" / "import the module first". No steer is better
  than a speculative one.
- **B5 — One diagnostic per root cause.** Downstream consequences of an
  already-reported failure stay silent (`Unknown`-typed uses do not
  cascade); sites inside re-running inference guard once-per-site (the
  `reported_literal_errors` pattern). A second diagnostic must add
  information, not repetition.
- **B6 — State the language rule when the construct is disallowed.** The
  model: "vilan has no const declarations — write `let x = const ..`".
  Prohibitions explain themselves and name the sanctioned spelling.

## 3. Mechanics

- **C1 — Deterministic.** Stable order, stable wording; no HashMap-order
  artifacts in which-diagnostic-fires-first or in rendered lists.
- **C2 — The pin is the qualification.** Every diagnostic class carries an
  `assert_fails_spanning` pin: exact span text + a message fragment. A
  site with no pin is unaudited by definition. (105 exist; the audit adds
  the rest per batch.)
- **C3 — Secondary notes.** A2 and B3 sometimes need a second location
  ("generated by `[derive(Wire)]` on `Point`", "first call here"). Today
  `Error` carries one span; the proposal adds an optional secondary
  `note: (Span, String)` rendered as an ariadne sub-label — one, not a
  list, to keep diagnostics terse. **Open question 1.**

## 4. The classes outside `diagnostics.push`

- **Lexer/parser errors** are chumsky-generated expected-lists ("found X,
  expected one of …15 tokens"). Curated parse messages are genuinely the
  handwritten parser's territory (H6); this arc records the worst
  offenders (the shift/`>` split, `!=` soup) with targeted `labelled`
  improvements only. **Open question 2.**
- **Warnings** (`must_use`, unused bindings) follow the same rules; the
  audit covers them in the same ledger.

## 5. The audit process

1. **Ledger.** Enumerate all 181 sites into `proposal/diagnostics-ledger.md`
   (site → class → verdict). Verdicts: QUALIFIES (pin cited), RE-ANCHOR
   (fails A-rules), REWORD (fails B-rules), DEMOTE (a cascade — delete or
   guard), NOTE-NEEDED (waits on C3's secondary note).
2. **Batches.** ~25 sites per slice, each slice suite-gated and committed
   with its new pins; the ledger updates in the same commit. Expected
   heavy spots, from memory: the E7 long tail (never re-audited), E8's
   async-rpc boundary case, macro-engine expansion failures.
3. **Exit criterion.** Every site has a ledger verdict and every QUALIFIES
   has a pin. The rules become CLAUDE.md-adjacent policy for new code
   (a new diagnostic cites the rules it satisfies).
